# Query — 위키를 근거로 답하고, 답을 다시 축적

이 스킬은 Andrej Karpathy의 **LLM Wiki** 원형을 따른다. 위키의 목적은 조회로 끝나지 않는다.
답을 합성하는 과정에서 나온 **새 통찰(비교·분석·연결)도 위키에 파일링**해서, 다음 질문이
더 적은 노력으로 풀리게 만드는 **복리 축적**이 핵심이다.

## 0. 먼저 schema.md를 읽는다 (필수 · 위반 금지)

작업 전 **반드시 저장소 루트의 `schema.md`를 Read** 하라. schema.md가 구조·규약·워크플로우의
**단일 기준(SSOT)**이다. schema.md의 "Query — 질문에 답하기" 절과 어긋나면 **schema.md를 따른다**.

## lazy-loading 원칙 (컨텍스트를 아껴라)

위키 전체를 로드하지 마라. 인덱스는 **2계층**이다:

1. 허브 `wiki/index.md` — 유형(overview/summary/entity/concept/decision/question)을 고른다.
2. 그 유형의 폴더 인덱스 `wiki/<type>/<type>-index.md` **만** 열어 관련 페이지를 좁힌다.

관련 페이지만 정밀하게 읽는 것이 원칙이다 — 필요하면 그 페이지가 링크하는 raw/까지만 확장한다.

## 실행 단계 (schema.md의 Query 워크플로우)

1. **드릴다운** — 허브 index → 해당 폴더 인덱스에서 관련 페이지를 찾는다 (전체 카탈로그 로드 금지).
2. **합성** — 그 페이지들(+필요 시 raw)을 읽고 답을 만든다. 결정에는 근거·기각 대안이,
   질문에는 status(open/answered)가 있으니 그대로 반영한다.
3. **축적** — 답이 재사용 가치가 있는 **새 지식**(여러 페이지를 잇는 비교·분석, 새로 드러난 미해결 질문)이라면
   해당 유형의 새 wiki 페이지로 파일링한다. 그러면 이후엔 `/ingest`의 인덱스·log 갱신 규칙을 따른다.
   - 여러 결정을 비교·정리 → 대개 `concept` 또는 새 `decision`.
   - 답하다 새 미해결 질문이 드러남 → `wiki/question/question-<slug>.md` (`status: open`).
   - 단순 사실 조회처럼 축적 가치가 없으면 파일링하지 않는다 (억지로 만들지 말 것).

## 근거 표기

- 답에는 근거가 된 페이지를 `[[wikilink]]`로 인용한다 (예: "야간 배치로 정한 이유는 [[decision-nightly-batch]] 참조").
- 위키에 답이 없으면 "위키에 근거 없음"이라고 명시하고, 필요하면 새 question 페이지로 남긴다 — 추측을 사실처럼 말하지 않는다.

## 완료 게이트 (끝내기 전 자가 점검)

- [ ] 허브 → 폴더 인덱스 순으로 드릴다운했다 (전체 로드하지 않았다).
- [ ] 답의 근거 페이지를 `[[wikilink]]`로 인용했다.
- [ ] 축적 가치가 있는 합성 결과는 새 페이지로 파일링하고, 인덱스·`log.md`를 갱신했다
      (`## [YYYY-MM-DD] query | <질문 요지>` + 생성/갱신 페이지).
- [ ] 파일링했다면 마무리로 `/lint` 실행을 권한다.
