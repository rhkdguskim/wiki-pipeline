# wiki-ops — 위키 작업 라우터 (오케스트레이터)

wiki_pipeline 지식 위키(Karpathy LLM Wiki)의 작업을 분류해 전문 에이전트로 위임하는 통합 스킬.
개별 스킬(ingest/query/lint)이 "어떻게"를 담는다면, 이 스킬은 "무엇을 누구에게 언제"를 정한다.

## 실행 모드: 서브 에이전트 (전문가 풀 패턴)

위키 작업은 사용자가 상황별로 하나씩 호출하는 **독립·순차** 작업이라 팀 상시 통신이 불필요하다.
라우터가 요청을 분류해 해당 에이전트를 `Agent` 도구로 호출하고 결과를 수집한다.

## 상태 모델 (위키 특화 — `_workspace/` 없음)

일반 하네스와 달리 중간 산출물 폴더를 만들지 않는다. **위키 저장소 자체가 상태**이고,
`wiki/log/`(날짜별 append-only)가 **감사 추적**이다. raw/는 불변 원본, wiki/는 갱신 대상.

## 에이전트 구성

| 에이전트 | subagent_type | 역할 | 스킬 | 산출물 |
|---------|--------------|------|------|--------|
| wiki-ingestor | wiki-ingestor | 새 지식을 raw/ 보존 + wiki/ 증류 | ingest | raw/·wiki/ 페이지·인덱스·log |
| wiki-librarian | wiki-librarian | 위키 근거 조회·합성·축적 | query | 답변 + (가치 시) 새 페이지·log |
| wiki-linter | wiki-linter | 무결성 점검·수정 | lint | lint 보고 + 수정 + log |

모든 `Agent` 호출은 `model: "opus"`로 지정한다.

## 워크플로우

### Phase 0: 컨텍스트 확인
1. `docs/schema/schema.md`를 Read 한다 (모든 위키 작업의 단일 기준).
2. 최근 이력을 확인한다: `grep "^## \[" wiki/log/*.md | tail -5`.
3. 후속 요청이면("아까 이어서", "다시") 직전 log 항목이 어떤 작업이었는지 파악해 이어붙인다.

### Phase 1: 작업 분류 (라우팅)
요청을 아래 의도로 분류한다. 여러 개면 **의존 순서대로** 나열한다 (대개 ingest → lint, query → lint).

| 의도 신호 | 라우팅 대상 |
|-----------|------------|
| 새 소스·결정·자료를 위키에 넣음 (반영/정리/추가) | wiki-ingestor |
| 위키 내용으로 질문에 답함 (찾기/왜/무엇) | wiki-librarian |
| 무결성·링크·인덱스 점검 (점검/검사/정합성) | wiki-linter |
| 반영/조회 **후 검증까지** 요청 | 해당 에이전트 → 이어서 wiki-linter |

분류가 모호하면 추측하지 말고 사용자에게 어떤 작업인지 한 번 확인한다.

### Phase 2: 에이전트 실행
분류 결과에 따라 `Agent` 도구를 호출한다. 전문가 풀은 대개 **순차 단일 호출**이다.

- 단일 작업: 해당 에이전트 1회 호출 (`run_in_background: false`, 결과를 사용자에게 요약).
- 연쇄 작업(예: 반영 후 점검): 앞 에이전트 완료 후 그 결과를 다음 에이전트 프롬프트에 요약해 전달하고 순차 호출.
- 호출 프롬프트에는 반드시 포함: (a) 사용자의 원 요청, (b) "docs/schema/schema.md를 먼저 읽고 해당 스킬을 Skill 도구로 실행하라", (c) 후속이면 직전 작업 맥락.

예시 (반영 + 점검 연쇄):
```
Agent(subagent_type="wiki-ingestor", model="opus",
      prompt="<소스> 를 위키에 반영하라. 먼저 schema.md를 읽고 ingest 스킬을 따르라.")
→ 완료 후
Agent(subagent_type="wiki-linter", model="opus",
      prompt="방금 ingest로 <건드린 페이지>가 갱신됐다. 무결성을 점검·수정하라. lint 스킬을 따르라.")
```

### Phase 3: 종합
1. 각 에이전트 반환값을 수집한다.
2. 무엇이 생성/갱신/수정됐는지, `wiki/log/<오늘>.md`에 어떤 항목이 남았는지 한 줄로 요약해 사용자에게 보고한다.
3. 연쇄에서 뒷 단계가 앞 단계의 결함을 발견하면 그 사실을 명시한다 (숨기지 않는다).

## 데이터 전달 프로토콜

| 전략 | 방식 | 용도 |
|------|------|------|
| 반환값 기반 | `Agent` 반환 메시지 | 에이전트 결과를 라우터가 수집 |
| 파일 기반 | wiki/(지식·log/)·raw/ (저장소 자체) | 실제 지식 산출물·감사 추적 |

중간 산출물 폴더(`_workspace/`)를 만들지 않는다 — 위키 저장소가 곧 산출물이다.

## 에러 핸들링

| 상황 | 전략 |
|------|------|
| 에이전트 1회 실패 | 1회 재시도. 재실패 시 어디까지 됐는지 log 기준으로 보고하고 중단 |
| 분류 모호 | 추측 금지 — 사용자에게 의도 확인 |
| 연쇄 중 앞 단계 실패 | 뒷 단계를 실행하지 않고 실패 지점을 보고 (부분 반영 상태를 명시) |
| 에이전트가 모순/중복 발견 | 삭제하지 말고 병기해 보고, 판단은 사용자에게 |
| raw 수정 시도 감지 | 즉시 중단 — raw는 불변 |

## 테스트 시나리오

### 정상 흐름 (연쇄)
1. 사용자: "이 설계 논의 정리해서 위키에 넣고 점검까지 해줘"
2. Phase 0: schema.md 읽기 + log 최근 이력 확인
3. Phase 1: ingest + lint로 분류 (순서: ingest → lint)
4. Phase 2: wiki-ingestor 호출 → 완료 후 wiki-linter 호출 (앞 결과 전달)
5. Phase 3: 생성/갱신 페이지 + lint clean 여부 요약 보고
6. 예상 결과: raw/ 1건 + wiki/ 페이지 갱신 + `wiki/log/<오늘>.md`에 ingest·lint 2항목

### 에러 흐름
1. 사용자: "위키 손봐줘" (의도 모호)
2. Phase 1에서 반영/조회/점검 중 무엇인지 분류 실패
3. 사용자에게 "새 내용 반영인가요, 조회인가요, 점검인가요?" 확인
4. 답변에 따라 정상 흐름으로 진입
