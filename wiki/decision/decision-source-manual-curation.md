# 결정: 등록은 운영자 수동 큐레이션이다 — 시스템은 방치를 자동 판정하지 않는다

대시보드에서 **사용할 프로젝트만 운영자가 골라 등록**한다. CI/릴리스가 없거나 오래 정체된 방치
소스(예: ros-codec)를 시스템이 자동 판정해 배제하는 로직은 두지 않는다. 등록 자체가 사람 판단 기반의
수동 큐레이션이다.

## 운영 모델 — "막는다"가 아니라 "안 쓸 건 등록 안 함"

- 방치/CI-less 소스를 "등록 불가로 시스템이 막는다"가 아니라, **쓸 프로젝트만 골라 등록한다**.
- 안 쓸 프로젝트는 애초에 등록되지 않으므로 파이프라인 대상이 아니다 — 배제 로직이 불필요.

## 함의 — 자동 판정 기준이 소거됨

- 방치/활성 **자동 판정 기준**(마지막 커밋·릴리스 시점 감지)은 **불필요** — [[question-ci-less-source-policy]]가 열어 두었던 "활성/방치 판정을 자동 감지할지" 검토 항목이 소거된다.
- 이 결정으로 [[question-ci-less-source-policy]]가 답을 얻는다 (매뉴얼이 MVP에 포함되면서 이 정책이 MVP 블로커였다 → [[decision-mvp-scope]]).

## 별개로 유지되는 것 — compare 404 자동 비활성화

브랜치·레포 **소실**(compare 404) 자동 비활성화([[decision-branch-loss-policy]])는 그대로 유지된다.
이건 **등록 후 소실** 처리라 등록 시점의 수동 큐레이션과 별개 관심사다 — 운영자가 골라 등록한 소스라도
나중에 브랜치가 사라지면 자동 비활성화된다.

## 기각 대안

- **방치 자동 판정·자동 배제** — 마지막 커밋/릴리스 시점으로 활성/방치를 시스템이 판정해 등록을 막거나 파이프라인을 건너뛴다. 사람이 어차피 쓸 프로젝트를 아는데 판정 로직을 얹으면 오탐·경계값 논쟁만 늘어 기각.
- **CI-less 소스에 정적만 적용(매뉴얼 제외) 자동 분기** — 소스별 파이프라인 대상은 등록 시 운영자가 명시하면 되므로 자동 분기 불필요.
