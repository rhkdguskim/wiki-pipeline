# 결정: 시나리오 세트의 소유·유지 주체는 과제 담당자 — 대시보드에서 정의·갱신

[[decision-hybrid-app-traversal]]의 결정적 뼈대인 시나리오 세트를 **과제 담당자**가
대시보드에서 직접 정의·유지한다.

## 세 가지

- **소유 = 과제 담당자** — 그 과제를 아는 사람이 시나리오를 만들고 앱 변화에 맞춰 갱신. 책임 소재가 과제에 귀속.
- **저장 = app 등록의 일부** — 시나리오 세트는 app 등록 정보의 일부로 서버 DB(SoT)에 저장 → [[entity-manual-pipeline]] · [[decision-db-source-of-truth]].
- **형식 = 대시보드 편집** — 코드가 아닌 대시보드 폼으로 편집. 형식 구체(필드·스텝 표현)는 구현 시 확정.

## 근거

- 시나리오는 앱 도메인 지식이 필요 — 중앙 팀이나 자동 생성보다 과제 담당자가 가장 잘 안다.
- 앱이 바뀌면 시나리오도 갱신돼야 한다 — 소유자가 명확해야 유지가 지속된다.
- 대시보드 편집은 코드 배포 없이 갱신을 허용 → 운영 민첩성.

## 기각 대안

- **기존 QA/테스트 자산 재사용** — 재사용성은 좋지만 (1) 형식 변환·동기화 부담 (2) 어느 과제에 무슨 자산이 있는지 불규칙. 있다면 담당자가 가져와 쓰는 형태로 보조.
- **시나리오 최소화 + 자율 탐색 위주** — 커버리지 보장([[question-ui-coverage-completeness]])이 약해진다.

이 결정이 [[question-scenario-set-ownership]]을 답한다. 시나리오가 커버하는 범위 + 탐색이 커버하는 범위의 합산 측정은 [[question-ui-coverage-completeness]]로.
