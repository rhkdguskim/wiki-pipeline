# sha 포인터 멱등성 패턴

**한 줄 요약: 소스 레포마다 "여기까지 문서에 반영했다"는 책갈피(`last_processed_sha`)를 두고, 배치가 끝까지 성공했을 때만 책갈피를 옮긴다.**

멱등성(idempotent) = 같은 작업을 몇 번 다시 돌려도 결과가 한 번 돌린 것과 같은 성질.
이 하나의 규칙이 **유실 방지 · 재실행 안전 · 디바운스**를 모두 해결한다.

## 규칙은 단 하나

```
성공: compare → 생성 → MR 생성 성공 → 완료 보고 → 그때서야 sha 전진
실패: 어느 단계든 실패 → sha 그대로 → 다음 run이 같은 구간을 다시 처리
```

## 시나리오로 보기 (X-LAB 레포)

| 밤 | 책갈피 | HEAD | 일어나는 일 |
|----|--------|------|------------|
| 월 | A | B (커밋 15개) | compare(A→B) → 문서 생성 → MR 성공 → **책갈피 = B** |
| 화 | B | C | 생성 중 AI API 장애 → **책갈피 B 그대로**, MR 없음 |
| 수 | B | D | compare(B→D) — 화요일 놓친 커밋까지 자동 포함 → 성공 → 책갈피 = D |

- 화요일 실패를 아무도 기억·복구할 필요 없다 — "책갈피가 안 움직였다"는 사실 자체가 미처리 기록
- 책갈피는 소스 단위 — X-LAB 성공·ROC 실패면 ROC만 재처리
- 재처리 시 열린 자동 MR을 **갱신** (새로 만들지 않음) → 중복 MR 없음 ([[decision-mr-review-gate]])

## 덤으로 얻는 것: 디바운스 공짜

compare API는 `from=책갈피&to=HEAD` 구간의 **최종 변경 파일 집합**을 준다.
커밋 100개가 같은 파일을 고쳤어도 결과는 "그 파일 1개" — 커밋마다 반응하는 큐·디바운스 코드가 통째로 불필요.
[[decision-pull-model]]이 성립하는 기술적 근거.

## 방어 로직 (책갈피가 어긋나는 경우)

- **신규 등록**: 초기 책갈피 = 등록 시점 HEAD (비우면 첫 배치가 레포 전체 히스토리를 "밀린 것"으로 오인)
- **force-push/rebase**: 책갈피가 현재 브랜치 조상인지 검증 → 무효면 "최근 N일" fallback + 경고. 예방: main protect
