# 2026-07-07 열린 질문 결정 세션

wiki-query로 열린 질문 목록을 추려 사용자에게 결정을 요청. 아래 4건 확정.

## 진행 방식

- query 워크플로우로 question 인덱스에서 open 상태 질문 13건을 추림.
- Phase 3+ 후보(테마 2차 확장·리뷰 피드백 루프·문서 Q&A/RAG)와 운영단계로 연기된 시크릿 저장 보안은 제외.
- 영향이 크고 지금 답 가능한 4건을 사용자에게 질의 → 확정.

## 확정 1 — MVP 절단선: 정적 + 매뉴얼 둘 다

**질문:** MVP 첫 출시 범위를 무엇으로 확정하는가? (위키 후보안: 정적 파이프라인 1개 + 최소 Control Plane)

**결정:** 첫 출시에 **정적(Docu-Automatic)과 매뉴얼 추출 파이프라인을 둘 다 포함**한다.

- 위키가 도출한 잠정 후보안(정적만)과 **다름** — 사용자가 매뉴얼 포함을 선택.
- 함의: 매뉴얼 파이프라인의 open 질문(아티팩트 타입 dispatch 등)이 MVP 블로커로 승격된다.
- 코드 인덱스는 여전히 시스템 범위 자체에서 제외(개인 관리 이관, 2026-07-06 확정).
- SCM 커넥터는 인터페이스 유지하되 구현은 GitLab 1개(실측 완료 환경). GitHub 커넥터는 MVP 이후.

## 확정 2 — 등록 baseline: A(null → 전체 초기화)

**질문:** 신규 등록 소스의 첫 문서화 baseline(last_processed_sha 초기값)은?

**결정:** **A. sha가 없으면(null) 전체 코드베이스를 모두 확인하여 initialize**한다.

- 위키 잠정안(A 기본 + 초기 전량 backfill을 정기 배치와 분리된 별도 개념으로)과 일치.
- 등록 시 last_processed_sha = null, 첫 처리는 전체 코드베이스를 확인해 문서를 initialize.
- "기존 코드 문서화"는 자동화의 핵심 가치이므로 B(등록 시점 HEAD, 변경분만)를 버린다.
- 방대한 역사가 야간 배치 한 밤에 터지지 않도록, 초기 전량 backfill은 대시보드 트리거 별도 작업(진행률 표시)으로 분리하고 정기 야간 배치는 증분만 처리하는 방향(구현 세부).

## 확정 3 — 방치 소스: 운영자 수동 큐레이션 (자동 판정 아님)

**질문:** CI/릴리스가 없는 방치 소스(예: ros-codec, 2018년 이후 정체)를 어떻게 처리하나?

**결정:** **대시보드에서 사용하지 않을 프로젝트는 운영자가 애초에 등록하지 않는다.** 시스템이 방치를 자동 판정해 배제하는 로직은 두지 않는다 — 등록 자체가 사람 판단 기반의 수동 큐레이션.

- "등록 불가로 시스템이 막는다"가 아니라 "쓸 프로젝트만 골라 등록한다"는 운영 모델.
- 따라서 방치/활성 자동 판정 기준(마지막 커밋·릴리스 시점 감지)은 **불필요** — 열린 검토 항목이 소거됨.
- 브랜치·레포 소실(compare 404) 자동 비활성화는 기존대로 유지(이건 등록 후 소실 처리라 별개).

## 확정 4 — 테마 경계: 독자 축으로 경계 명시

**질문:** requirements 테마와 dev-guide 테마가 소재(런타임·의존성)가 겹친다. 어떻게 가르나?

**결정:** **통합하지 않고 독자 축으로 경계를 명시**한다.

- requirements = 소스 빌드 없이 제품을 설치·실행하는 환경/조건 (독자: 설치자·운영자)
- dev-guide = 소스에서 빌드·테스트·디버그하며 개발하는 환경 (독자: 개발자)
- 겹치는 사실은 상세를 한쪽에 두고 다른 쪽이 참조.
- 경계 문장을 테마 정의(프롬프트)에 명시할지 critic 적합성 기준에 반영할지는 구현 세부(양쪽 다 가능).
- decision-manual-taxonomy-two-reader와 같은 원리(독자 축 분리).
