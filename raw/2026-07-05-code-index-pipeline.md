# 코드 인덱싱 파이프라인 설계 논의 (2026-07-05)

> Claude Code 세션 논의 기록. 사용자 요청 원문 요지와 Q&A로 확정된 사항을 보존한다.

## 사용자 요청 (원문 요지)

1. "codegraph같은 기능도 제공을 할거야. 근데 codegraph의 기술적인 의존이 있으면 안되고,
   우리만의 인터페이스를 설계해서 codegraph의 구현은 숨겨야 해. 나중에 다른 기술로 갈아 끼울 수 있는
   구조로 갈 거야. 하나의 파이프라인이 더 생기는 거지 — codegraph같이 코드를 인덱싱해서 저장해주는 파이프라인이야."
2. "codegraph의 의존성이 없게 우리 레이어의 인터페이스를 설계해서 구현해야 해.
   code traversal이 가능하도록 구성되어야 합니다."

## Q&A로 확정된 사항

| 질문 | 답 |
|------|---|
| 인덱싱 입력 | GitHub·GitLab에 등록된 레포 내용 (문서 파이프라인과 같은 등록 체계) |
| 인덱스 소비자 | **개발자 직접 조회** (코드 검색·탐색) |
| 신선도 | "git commit 시에 계속 돌면 좋을 것 같다. 이 기능은 AI 기능을 활용하지 않고 속도도 빠르다" — 야간 배치 아님 |
| commit 감지 방식 | **짧은 주기 폴링** 채택 (webhook·소스 레포 CI job 기각) |
| query 최소 요구 | **code traversal** (정의↔참조·호출·의존 관계 순회) 1급 연산 |

## 논의 중 도출된 정리

- 기존 pull/야간 배치 결정이 "push 즉시 실행"을 기각한 근거는 전부 AI 워크로드 전제(호출 비용·러너 부하·MR 리뷰 폭주)
  → 비-AI·빠름·MR 없음인 인덱싱에는 적용되지 않는다. **야간 배치는 시스템 전역 속성이 아니라 문서 파이프라인의 속성.**
- "소스 레포 무수정" 원칙은 유지 → CI job 방식 계속 기각. webhook은 GitHub(외부 클라우드) → 사내 서버
  인바운드 개방이 필요해 기각, 짧은 주기 폴링 채택 (pull 메커니즘을 주기만 바꿔 재사용).
- 쓰기 경로(폴링 → 인덱싱)와 읽기 경로(개발자 질의)가 분리된다. 재인덱싱 중에도 직전 인덱스로 질의 가능해야 한다.
- sha 포인터는 파이프라인별 독립 — 문서 sha는 MR 성공 기준, 인덱스 sha는 인덱싱 성공 기준. 한쪽 실패가 다른 쪽을 막지 않는다.
- 산출물은 MR이 아니라 인덱스 저장소 — 파생 데이터, 사람 리뷰 게이트 없음, 언제든 재구축 가능.
