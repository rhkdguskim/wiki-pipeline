# 파이프라인 실시간 모니터링 요구사항 (2026-07-05)

> raw source — 불변. cross-cutting 설계 지시.

## 지시 (원문 취지)

지금까지 설계한 파이프라인은 3가지:
1. 정적 문서 파이프라인 (코드 diff → 기술문서, Docu-Automatic 엔진)
2. 매뉴얼 추출 파이프라인 (릴리스 앱 관측 → 사용자/엔지니어 매뉴얼)
3. 코드 인덱스 파이프라인 (소스 인덱싱 → 개발자 조회)

이렇게 **계속 추가되는 모든 파이프라인**은 대시보드에서 **진행상황이 실시간으로 모니터링**되어야 한다.
파이프라인을 개발할 때 모니터링이 되도록 처음부터 잘 고려한다.

## 함의

- 사후(완료) 보고만으론 부족 — **실행 중 진행상황**(단계·진척)이 실시간으로 보여야 함.
- 파이프라인이 이기종(AI 무거움 / MCP 구동 / 비-AI 폴링)이라도, 지휘 계층(관리 서버 대시보드)이
  **공통 형태로 집계·표시**할 수 있어야 함 → 공통 관측성 계약 필요.
- "계속 추가됨" → 신규 파이프라인도 같은 계약을 따르도록, 모니터링을 파이프라인 설계의 1급 제약으로 둔다(사후 부착 금지).
- 관리 서버는 이미 이력 DB(SoT) + 대시보드를 가지므로 자연스러운 귀결. 기존 완료 보고·pipeline webhook 계약을
  실시간 진행으로 확장.
