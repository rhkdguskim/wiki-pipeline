# 문서 라이프사이클 diff

**한 줄 요약: 기존 문서 집합을 "현재 실제 상태"와 대조해 더할 것·고칠 것·지울 것으로 가른다.**

문서 자동화는 매번 백지에서 다시 쓰는 게 아니라, 이미 있는 문서를 대상 상태에 맞춰 **증분 갱신**하는
문제다. 세 가지 변화가 있다: 추가(ADD)·수정(UPDATE)·삭제(DELETE).

## 삭제가 가장 어렵다

ADD·UPDATE는 "지금 있는 것"만 보면 대체로 판정된다. 그러나 DELETE는 "없어졌다"를 확신해야 하는데,
현재 상태에서 무언가가 **안 보이는 것**은 두 가지다 — 정말 제거됐거나, 관측이 놓쳤거나. 그래서 삭제는
**현재 상태 관측만으로는 안전하지 않고**, "제거됐다"는 별도 신호(예: 변경 이력)와 교차해야 한다.

## 안전한 삭제의 일반형

```
DELETE = (변화 이력이 '제거'라고 말함) AND (현재 상태에서도 부재)
```

두 독립 신호가 함께 "없음"을 가리킬 때만 삭제하고, 파괴적 작업이므로 사람 확인 게이트를 덧댄다.

## 왜 개념인가

이 라이프사이클은 어떤 문서 파이프라인에도 적용된다 — 신호가 코드 diff든 UI 관측이든,
"기존 ± 현재 = 변경분"이라는 뼈대는 이식성 있다.
