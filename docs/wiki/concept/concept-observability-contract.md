# 파이프라인 관측성 계약

**한 줄 요약: 이기종 파이프라인이 공통 형태의 진행 이벤트를 지휘 계층에 보고하고, 지휘 계층이 이를 집계·표시한다.**

여러 종류의 작업(무거운 AI 생성, 원격 UI 구동, 가벼운 폴링)이 한 대시보드에서 나란히 모니터링되려면,
각자 내부가 아무리 달라도 **바깥으로 내보내는 진행 신호의 형태는 같아야** 한다.

## 계약의 뼈대

- 파이프라인은 **진행 이벤트**를 낸다: 시작 → 단계 전이 → 진척(예: N/M) → 완료/실패.
- 지휘 계층은 이 이벤트를 받아 **집계·표시**하고, 실행 이력(SoT)에 남긴다.
- 워커는 "무엇을 하는지"만 다르고, "어떻게 보고하는지"는 동일 계약을 따른다.

## 실시간 ≠ 사후

완료(terminal) 보고만으로는 "돌고 있다/끝났다"만 안다. 실시간 관측성은 **실행 중 진척**을 요구한다 —
긴 작업일수록 "지금 어디쯤"이 운영·디버깅의 핵심이다.

## 왜 개념인가

이 계약은 어떤 오케스트레이션에도 적용된다 — 워커가 문서 생성기든 인덱서든, "이기종 워커 + 공통 진행 계약
+ 중앙 집계"라는 뼈대는 이식성 있다. 계약을 처음부터 두면 새 워커를 붙일 때 모니터링이 공짜로 따라온다.
