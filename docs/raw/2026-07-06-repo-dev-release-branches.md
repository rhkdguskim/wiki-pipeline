# 등록 모델 전환 — 레포 1개 + 개발/배포 브랜치 (2026-07-06)

> 사용자 지시로 등록 모델을 바꾼 기록. 원본 불변 — 위키에서 증류.
> 앞선 "레포×브랜치 1개 = 등록 원자단위" 모델([[decision-repo-registration-flow]] 초판·
> [[decision-branch-loss-policy]] 정책 C)을 대체한다.

## 사용자 지시

> "다시 레포지토리는 하나만 등록하도록 하고 개발브랜치, 배포브랜치를 등록할 수 있도록 하는 흐름으로 가져가자"

## 전환 내용

### 이전 모델 (폐기)

- 등록 원자 단위 = **레포 × 브랜치 1개**. 같은 레포를 다른 브랜치로 문서화하려면 등록을 여러 번.
- "한 등록에 여러 브랜치 묶기"를 기각했었음.
- 브랜치 등록 정책 C(모든 브랜치 노출 + 비-기본 경고, 아무 브랜치나 다중 등록).

### 새 모델 (확정)

- **등록 원자 단위 = 레포 1개.** 레포당 등록은 1건.
- 그 등록 안에 **개발 브랜치 1 + 배포 브랜치 1 (고정 2)** 을 지정한다.
- **역할이 문서 산출을 가른다** (grill-me 확정):
  - **개발 브랜치** → 최신 기술문서. 자주 갱신(compare 기반, 야간 배치).
  - **배포 브랜치** → 안정 릴리스 문서·매뉴얼. 태그/릴리스 트리거.
  - 두 브랜치가 docs-hub에서 **다른 출력물**로 간다.
- **브랜치 개수 = 개발 1 + 배포 1 고정** — 역할별로 정확히 하나씩. UI가 명확해지고 doc_dir도 결정적.

### docs-hub 폴더 구조 (전환)

- 이전: 등록(=레포×브랜치)마다 `full_namespace_path / branch` **평면 폴더**.
- 새: **레포 1폴더 + 브랜치 하위폴더** — `full_namespace_path/` 아래 `dev/`·`release/`.
  한 레포의 문서가 한 곳에 모여 탐색이 자연스럽다.

## 연쇄로 바뀌는 것

- **sources 테이블 키** — (project_id, branch) 행 다수 → **project_id 단위 1행 + dev/release 브랜치 2컬럼**
  (또는 등록 1행에 두 브랜치 참조). doc_dir도 레포 단위 + 역할 하위경로.
- **compare/트리거** — 두 브랜치를 독립 추적하되 **한 등록이 소유**. 개발=compare 야간,
  배포=릴리스/태그 트리거.
- **브랜치 소실(404) 처리** — 여전히 유효하나 이제 "등록 안의 두 브랜치 각각"에 적용.
  개발·배포 둘 다 대개 protected/안정이라 재활성화 분기의 protected 경로가 주로 쓰임.

## 미확정 (후속 grilling 대상)

- 개발/배포 브랜치 중 **하나만 지정**(예: 배포만 있고 개발 없음)을 허용할지 — 필수 2개 vs 선택.
- 배포 브랜치가 **여러 변종**(pcc30류)일 때 개발 1+배포 N을 열지 — 이번엔 고정 2로 확정, 확장은 후순위.
