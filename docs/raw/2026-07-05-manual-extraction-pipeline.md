# 사용자/엔지니어 매뉴얼 추출 파이프라인 설계 논의 (2026-07-05)

> raw source — 불변. 세션 논의 + 결정 기록. Docu-Automatic(코드→문서)과 **별개**인 신규 파이프라인.

## 요청 (원문 취지)

사용자/엔지니어 매뉴얼 자동 작성 파이프라인이 필요하다. 주 애플리케이션은 Windows Application이며,
레포의 csproj/vsproj를 "app"으로 등록해 사용한다. 원격제어 MCP(사내 제공)를 띄워 앱을 직접 구동하고,
그 MCP를 통해 UI의 모든 기능을 돌며 매뉴얼을 추출하는 파이프라인을 만들고자 한다.

## 확정 결정 (Q&A)

| 갈림길     | 선택                                        |
| ------- | ----------------------------------------- |
| MCP 제어면 | UI 자동화 인식 기본, 좌표 기반 데스크톱 제어 fallback      |
| 주행 전략   | 하이브리드 — 핵심 시나리오(결정적 뼈대) + 자율 탐색(완전 커버 보완) |
| 실행 시점   | 릴리스/버전 태그 시 (코드 커밋마다 아님)                  |
| 산출물     | docs-hub 통합 + MR 게이트                      |

## 정정 (2차 논의)

1. **빌드 안 함** — 빌드는 기존 CI가 담당, 릴리스가 아티팩트를 생성 → 최신 버전 아티팩트를 가져와 소비.
2. **MCP가 배포까지 담당** — MCP 파일전송으로 아티팩트를 앱 실행 환경에 전송·배치 → 앱 실행 → UI 전수 순회.
3. **commit 히스토리 기반 라이프사이클** — 기존 완성 매뉴얼 대비 추가/수정/삭제를 커밋 히스토리 분석으로 판단.
4. **Docu-Automatic과 별개** — 코드→문서와 다른 기능. 공유는 관리 서버·docs-hub·MR 게이트뿐, 생성 엔진은 신규.
5. **실행 기반은 사내 Windows CI** — 기존 파이프라인도 Windows CI에서 구동. Windows 앱 구동·UI 자동화가 가능한
   기반이 이미 존재(신규 인프라가 아니라 재사용).

## 파이프라인 (7단계)

1. 아티팩트 수집 — 릴리스 태그 → 그 버전 아티팩트 획득 (외부 빌드 CI 산출물 소비)
2. 배포 — MCP 파일전송으로 앱 실행 환경에 아티팩트 전송·배치
3. 실행 — 그 환경에서 앱 기동 (인증·데이터 등 전제 충족)
4. 전수 순회 — MCP UI 자동화로 모든 기능 순회(하이브리드). 화면·컨트롤·전이·캡처를 관측 로그로 수집
5. 매뉴얼 생성 + 라이프사이클 diff — 관측 + 커밋 히스토리 → 기존 매뉴얼 대비 add/update/delete
6. 제출 — docs-hub 브랜치 + MR → 사람 리뷰
7. 보고 — MR 성공 후에만 last_documented_version 전진 (멱등성)

## 매뉴얼 라이프사이클 판정 (두 신호 결합)

- 커밋 히스토리(last_documented_version ~ 태그): 무엇이 추가/제거/변경됐나. **삭제 탐지에 필수.**
- UI 전수 순회 관측: 지금 앱에 실제로 무엇이 있나(현재 상태 ground truth).
- ADD = 히스토리 신규 + 관측 신규 + 기존 매뉴얼에 없음
- UPDATE = 히스토리 변경 + 관측이 기존 매뉴얼과 불일치
- DELETE = 히스토리 제거 + 관측 부재 (두 신호 모두 "없음"일 때만) → MR 삭제 제안(자동 삭제 아님)

## app 등록이 갖는 것

소스 레포(커밋 히스토리) + 아티팩트 출처(릴리스) + 기동 프로파일·시나리오 세트 + last_documented_version.
기존 `source` 등록(compare API용)과 별도의 등록 종류.
