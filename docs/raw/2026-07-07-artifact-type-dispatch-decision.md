# 2026-07-07 아티팩트 타입 dispatch 결정 세션

MVP에 매뉴얼 파이프라인을 포함(→ 2026-07-07 MVP 절단선 결정)하면서 블로커로 승격된
`question-artifact-type-dispatch`를 grilling해 3개 하위 결정을 확정.

배경: 아티팩트 소비 경로는 Generic Package Registry로 이미 확정(decision-artifact-consumption,
실측 2026-07-06-wish-gitlab-api-survey). 그 위에서 소스별 타입 분기가 열려 있었다.

## 확정 1 — 구동 대상 타입: exe/msi만 (container·nuget 제외)

**질문:** generic(exe/msi)·nuget·container 중 무엇을 "구동 가능한 앱"(매뉴얼 대상)으로 보나?

**결정:** **exe/msi(Windows 설치본)만** 매뉴얼 대상으로 삼는다. container도 제외.

- nuget = 라이브러리성 자산이라 UI가 없어 구동·관측 불가 → 매뉴얼 대상 아님(자연 제외).
- container = 별도 런타임 환경 부담 → MVP에서 제외, 이후로 미룸.
- 함의: container로만 배포되는 소스(예: MCP 이미지)는 MVP 매뉴얼 대상에서 자연히 빠진다.
  이는 방치 소스 큐레이션(담당자가 등록 안 함)과 같은 결 — "MVP 매뉴얼 대상 밖".

## 확정 2 — 자산 선택: 담당자가 대시보드에서 지정

**질문:** 한 릴리스에 자산 11개(여러 exe·msi·설치기)일 때 어느 자산을 문서화 대상으로 고르나?

**결정:** **과제 담당자가 app 등록 시 대시보드에서 구동할 설치 자산을 지정**한다(파일명 패턴/명시 선택).

- 시스템이 규칙으로 자동 판별하지 않는다(릴리스마다 자산 명명이 달라 오판 위험).
- decision-scenario-owner-dashboard(시나리오 세트 담당자 정의)·시크릿 등록과 같은 축 — 담당자 수동 큐레이션.

## 확정 3 — MCP 기동 범위: 전송 + 설치 실행까지

**질문:** 원격제어 MCP(entity-remote-control-mcp)가 배포·기동을 어디까지 처리하나?

**결정:** MCP가 **아티팩트 전송 + 설치 스크립트/설치기 실행(silent install)까지** 수행해 앱을 구동 상태로 만든다.

- 파이프라인이 "실행 중인 앱"을 관측하려면 설치까지 자동화돼야 하므로 MCP가 설치 실행을 포함.
- MCP는 파일전송 + UI 자동화 원자 동작 제공(기존 정의) 위에, 설치 실행 오케스트레이션을 얹는다.

## 정합성 메모

세 답이 이번 세션의 일관된 방향을 따른다:
- 시스템이 자동 추론하지 않고 담당자가 대시보드에서 명시(자산 선택).
- MVP는 Windows 설치본(exe/msi)에 집중, container는 이후.
- container 제외는 MVP 매뉴얼 포함 결정 및 방치 소스 큐레이션과 같은 결로 처리.
