# 2026-07-06 — 코드 인덱스: 파이프라인 범위 제외 (개인 관리 이관)

> 대화 기록 (2026-07-06). 발단: 커밋 되돌리기 시 코드 인덱스 불일치 질문 → 논의 끝에
> 코드 인덱스를 중앙 파이프라인 범위에서 제외하기로 결정. 이 파일은 불변 원본이다.

## 1. 발단 질문 — 커밋 되돌리기와 인덱스 불일치

**사용자**: "개발자가 코드 그래프를 사용하게 될 때 사용자의 레파지토리의 커밋을 되돌리게 되면
코드 인덱스와 일치하지 않는 경우가 발생할 텐데 이건 어떻게 해결해야 할까?"

**어시스턴트 분석** (당시 유효하던 위키 결정 기반):

- 인덱스는 "현재 상태"가 아니라 "레포@sha의 스냅샷"(당시 decision-code-index-versioning) —
  불일치는 오류 상태가 아니라 버전 차이로 환원된다. 폴링 랙은 구조적으로 항상 존재.
- `git revert` = 새 커밋으로 히스토리 전진 → 기존 폴링 경로(compare → index → sha 전진)가
  특수 처리 없이 소화. 이미 해결된 케이스.
- `git reset --hard` + force-push = 히스토리 재작성 → 책갈피 조상 검증으로 감지(concept-idempotent-sha).
  복구는 문서 파이프라인("최근 N일" fallback — AI 비용·MR 소음 때문)과 달라야 한다: 코드 인덱스는
  구간(diff)이 아니라 **최종 상태만 소비**하고 비-AI·저비용이므로, "현재 HEAD 기준 전체 재인덱싱 →
  새 버전 스냅샷 → 원자 교체 → 책갈피=HEAD"가 정답. 델타 역산 불필요.
- 남는 요구사항: MCP 질의 응답에 "이 응답은 레포@sha X 기준" 메타데이터 노출 — 되돌리기만이 아니라
  폴링 랙 전반을 덮는 일반 해법.
- 로컬에서만 되돌리고 push하지 않은 경우: 중앙 인덱스는 등록된 원격 브랜치를 추적하므로 불일치가
  정상이며, sha 메타데이터가 그 사실을 소비 측에 드러낸다.

## 2. 결정 — 코드 인덱스는 개인이 관리, 파이프라인에서 제외

**사용자**: "code indexing은 개인이 관리하는 편이 나을 거 같아서 파이프라인에서 빼도록 하자.
그리고 다른 파이프라인 돌릴 게 있는지 아이디어를 알려줘."

- 코드 인덱스를 중앙 파이프라인 범위에서 **제외**한다. 개발자 개인이 로컬 도구로 관리한다.
- 1의 논의가 보여주듯, 중앙 인덱스는 원격 동기화 문제(신선도 랙 · 되돌리기/force-push 복구 ·
  push 안 된 로컬 상태와의 괴리)를 구조적으로 안고 간다. 개인 로컬 인덱스는 개발자의 작업 트리를
  직접 추적하므로 이 문제들이 발생 자체를 하지 않는다.
- 2026-07-05 확정된 코드 인덱스 결정군 8건(파이프라인 도입 · 프로바이더 추상화 · MCP 서빙 ·
  러너 git clone · 버전 스냅샷 · cg-colby 어댑터 · 단일 레포 범위 · 저장 평면)은 일괄 번복된다.
- 도구 조사(entity-codegraph의 cg-colby·cgc 실측)는 개인 도구 선택의 참고 자료로 유효하다.
- "다른 파이프라인 아이디어"는 브레인스토밍 요청 — 결정이 아니며 별도 논의로 이어진다.
