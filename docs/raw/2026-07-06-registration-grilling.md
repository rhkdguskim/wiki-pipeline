# 레포 등록·docs-hub 구조 grilling (2026-07-06)

> `/grill-me` 세션. 레포 등록 시나리오(decision-repo-registration-flow ingest 직후)의
> 설계 구멍·모호한 비즈니스를 사용자에게 질의해 확정한 기록. 원본 불변 — 위키에서 증류.
> 이 문서는 사용자와의 인터뷰 문답을 요약 보존한다(개별 발화 순서·표현은 압축).

## 배경

앞선 실측 조사([[2026-07-06-wish-gitlab-api-survey]])와 등록 시나리오 결정
([[decision-repo-registration-flow]])이 "등록 1건 = 레포별 project access token + 브랜치 1개"를
확정한 상태. 이 위에서 "등록된 뒤 무엇을 하는가"의 남은 모호함을 grilling으로 좁혔다.

## 확정된 결정

### 1. docs-hub 구조 — 소스 N개 → 레포 1개, 소스별 폴더 분리

- 모든 과제 문서를 **하나의 docs-hub 레포**에서 관리한다. 소스가 늘어도 문서 레포는 하나.
- 소스(= 레포×브랜치 등록 1건)마다 docs-hub 안에서 **폴더로 분리**된다.
- 근거: 파이프라인 로직·문서 사이트가 한 곳에 모여 유지보수 지점이 단일. (기존 entity-docs-hub의
  "과제별 Docusaurus instance"를 소스 입도의 폴더 규칙으로 구체화)

### 2. doc_dir 자동 생성 규칙

- 등록 시 사람이 폴더 경로를 입력하지 않는다 — **규칙으로 자동 생성**한다.
- 규칙: **`full_namespace_path / branch`**.
  - `full_namespace_path` = GitLab 프로젝트의 전체 네임스페이스 경로(그룹/서브그룹/프로젝트).
  - **브랜치를 항상 붙인다** — 같은 레포를 다른 브랜치로 여러 번 등록할 수 있으므로(아래 3 참조),
    브랜치를 빼면 폴더가 충돌한다.
  - 브랜치명에 **슬래시(`/`)가 있으면 `-`로 치환**한다(예: `release/1.2` → `release-1.2`).
    안 그러면 의도치 않은 폴더 깊이가 생긴다.
- 이 규칙은 `sources.doc_dir` 컬럼([[decision-db-source-of-truth]])의 값을 채운다.

### 3. 브랜치 등록 정책 = C (모든 브랜치 노출 + 비-기본 경고)

- 등록 UI는 `GET /projects/:id/repository/branches`의 **모든 브랜치를 노출**한다(제한 없음).
- 단, **비-기본(non-default) 브랜치를 고르면 경고**를 표시한다(안정성 낮을 수 있음 주의).
- 이는 [[decision-repo-registration-flow]]의 "브랜치 1개 = 등록 원자 단위"와 정합한다 —
  등록 1건은 여전히 브랜치 1개지만, **같은 레포를 다른 브랜치로 여러 번 등록**하는 것은 허용된다
  (각 등록이 독립된 원자 단위 = 독립 doc_dir 폴더). pcc30(브랜치 125개 태그·11 브랜치)처럼
  변종이 많은 레포는 필요한 브랜치를 각각 별도 등록한다.
- 후보 A(기본/protected만 허용)·B(기본만) 대비 C를 택한 이유: 사용자가 다중 브랜치 문서화를
  명시적으로 원했고, 경고로 위험을 알리되 선택 자유를 막지 않는다.

### 4. 좀비 등록 처리 = B (자동 비활성화 + 알림, 삭제 아님)

- 등록된 소스의 야간 compare가 **404**(브랜치·레포 소실)를 반환하면 = "좀비 등록".
- 처리: **자동 비활성화**(`sources.enabled = false`) + **대시보드 알림**. **삭제하지 않는다.**
- 근거: run_items 이력·등록 메타를 보존해야 감사 추적이 끊기지 않는다
  ([[decision-db-source-of-truth]]는 이미 "run_items 삭제 안 함, 비활성화 후에도 이력 보존").
- 5xx·타임아웃은 브랜치 소실이 아니므로 재시도(비활성화 아님).

### 5. 브랜치 소실 정책 — protected 플래그로 재활성화 분기

- **protected/default 브랜치**: 첫 404 즉시 비활성화(a) + 브랜치 복귀 시 **자동 재활성화**(d).
  안정 브랜치라 진동(flapping) 위험 없음.
- **비-protected 브랜치**(feature류): 첫 404 즉시 비활성화(a) + 복귀는 **수동 재활성화만**.
  자동 재활성화를 빼서 flapping·감사 단절을 차단.
- GitLab이 API로 알려주는 `protected` 플래그로 시스템이 안정/불안정을 실제 구분한다 —
  정책 3(C, 아무 브랜치 허용)의 유연성과 "안정 브랜치 가정"이 이 분기로 정합한다.
- 모든 비활성화는 대시보드 알림. 등록은 어느 경우도 자동 삭제하지 않는다(위 4와 일관).

## 아직 미확정 (grilling 계속 대상)

### baseline — 신규 등록 소스의 첫 문서화 기준점 (질의 중, 확정 아님)

`sources.last_processed_sha`의 등록 시 초기값을 무엇으로 둘지:

| 초기값 | 첫 배치 동작 | 트레이드오프 |
|--------|-------------|-------------|
| A. null(레포 최초 커밋) | 브랜치 전체 역사 문서화 | 완전하나 첫 실행이 폭발적(x-lab 8년치) |
| B. 등록 시점 HEAD | 등록 후 변경분만 | 가볍지만 기존 코드는 영원히 미문서화 |
| C. 사람이 baseline 지정(태그/sha) | 최근 릴리스부터 등 | 유연하나 등록이 무거워짐 |

- 제안(미승인): **A 기본 + "초기 전량 backfill"을 정기 야간 배치와 분리된 별도 개념으로**.
  등록 시 `last_processed_sha = null`, 첫 처리는 대시보드 트리거 명시적 backfill 작업(진행률 표시),
  정기 배치는 증분만. 이유 — "기존 코드 문서화"는 자동화의 핵심 가치라 B는 그걸 버림. 하지만 A를
  야간 배치에 그냥 태우면 8년치가 한 밤에 터지니 backfill을 1급 개념으로 분리.
- **사용자 최종 승인 전** — 위키에는 아직 결정으로 반영하지 않는다(질문 페이지로만 남길 수 있음).
