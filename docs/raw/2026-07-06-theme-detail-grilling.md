## 문답 (선택 ✔ · 기각 대안 병기)

1. **api-protocol 적용 대상(백엔드 소스) 판정**
   - ✔ 테마 체크리스트 (추천안): 소스 등록/설정 화면에 테마 체크리스트 — 기본 5개(기존 4 + dev-guide)
     자동 on, api-protocol은 담당자 opt-in. 향후 테마 추가에도 재사용하는 일반 메커니즘
   - 기각: 소스 성격 분류 필드(성격→테마 세트 자동 결정 — 간접 구조), scout 자동 판단(비결정적·비용·누락 위험)

2. **dev-guide 산출물 형태·포함 범위**
   - ✔ 1문서 · 코드 근거만 (추천안): ①환경 구성(사전 요구·의존·빌드) ②실행·테스트·디버그 ③프로젝트 구조.
     근거는 빌드 스크립트·CI 설정·매니페스트·README. 조직 규칙(코딩 컨벤션·브랜치 전략·리뷰 규칙)은 제외(환각 위험)
   - 기각: 조직 규칙 포함(담당자 입력 자료 필요), 2문서 분리(테마=문서 매핑 계약 변경)

3. **api-protocol이 다루는 통신 계약의 경계**
   - ✔ 외부 노출 API만: HTTP/gRPC 등 외부에서 호출하는 인터페이스만 문서화
   - 기각: 외부 API + 내부 프로토콜(Akka 메시지·소켓/장비 프로토콜 포함 — 추천안이었으나 기각),
     담당자 지정 범위(등록 부담·누락 공백)

4. **dev/release 브랜치 산출 라우팅**
   - ✔ 둘 다 dev만: 두 테마 모두 개발 브랜치 산출(dev/)에만 포함, release/는 기존 4테마 유지
   - 기각: dev-guide는 dev·api는 둘 다(추천안이었으나 기각), 둘 다 양쪽 포함

5. **환각 방지 검증 수준**
   - ✔ critic에 근거 대조 추가 (추천안): 두 테마 한정 — 문서의 모든 명령·설정·엔드포인트는
     source_files 내 실존 근거 필수, 없으면 FAIL. 기존 재시도(최대 2회) 사이클 재사용
   - 기각: 기존 critic 그대로(MR 리뷰 의존), 결정적 외부 검증(OpenAPI 대조·dry-run — 비용, Phase 3+ 후보)

6. **시크릿·사내 정보 기재 정책**
   - ✔ 시크릿 금지 + critic 검사 (추천안): 토큰·패스워드·자격증명 값 기재 금지 — 환경변수 이름으로만
     언급. critic 검사 항목에 시크릿 패턴 검출 추가. 사내 URL·피드 주소는 허용(docs-hub 사내 전용)
   - 기각: 제약 없음(시크릿 복제·회전 후 잔존 위험), 사내 정보 전면 마스킹(문서 효용 급감)
