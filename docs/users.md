# 사용자 · 핵심 유스케이스

← [문서 인덱스](./README.md)

## 사용자 유형

| 유형 | 누구 | 시스템과의 접점 |
|------|------|----------------|
| **운영자** (문서 관리자) | 초기: 본인 1인 | 소스 등록/해제, 수동 트리거, 대시보드 모니터링, 실패 대응 |
| **과제 개발자** | 각 과제 팀원 | 자동 생성 문서 MR 리뷰·머지 (시스템을 직접 조작하지 않음) |
| **문서 소비자** | 신규 팀원, 설치자/운영자, 개발자 | Docusaurus 사이트 열람 (테마별 대상 독자와 일치 → [generation](./features/generation.md)) |

## 핵심 유스케이스

- **UC1 — 야간 자동 갱신**: 운영자는 아무것도 하지 않는다. 밤 사이 배치가 돌고, 아침에 과제 개발자가 문서 MR을 리뷰·머지한다 → [scheduling](./features/scheduling.md)
- **UC2 — 신규 과제 온보딩**: 운영자가 대시보드(또는 API)에서 project_id·doc_dir을 등록한다. 소스 레포는 건드리지 않는다 → [sources](./features/sources.md)
- **UC3 — 수동 재생성**: 특정 과제 문서가 이상하거나 즉시 갱신이 필요할 때, 운영자가 해당 소스만 골라 트리거한다 (full 재생성 옵션 포함) → [scheduling](./features/scheduling.md)
- **UC4 — 실패 확인·복구**: 아침에 대시보드에서 어젯밤 run 결과를 확인한다. 실패한 소스는 원인(네트워크/토큰/엔진)을 보고 재트리거한다. 변경분은 유실되지 않는다 → [monitoring](./features/monitoring.md), [nfr](./nfr.md)
- **UC5 — 문서 열람**: 문서 소비자는 Docusaurus 사이트에서 과제별 문서를 본다. `auto_generated_warning` 태그가 있는 문서는 검증 미통과 상태임을 인지할 수 있다
